# Основные подходы к реализации rate limiter

### 1. Token Bucket Algorithm

*    Представляет "ведро" с фиксированным количеством токенов
*    Каждый запрос требует один токен
*    Токены пополняются с постоянной скоростью
*    Если токенов нет, запрос блокируется или отклоняется
*    Преимущества: простая реализация, справедлива к пакетным запросам
*    Недостатки: требует точного выбора размера бакета

### 2. Leaky Bucket Algorithm

*    Похож на token bucket, но запросы обрабатываются с постоянной скоростью
*    Как вода, вытекающая из ведра с одной скоростью
*    Любые "лишние" запросы отклоняются
*    Преимущества: контролирует пиковые нагрузки равномерно
*    Недостатки: менее гибкий при обработке всплесков

### 3. Sliding Window Log

*    Отслеживает временные метки каждого запроса в памяти
*    Для каждого нового запроса проверяет логи в текущем окне времени
*    Если количество запросов превышает лимит, блокирует его
*    Преимущества: точное соответствие лимиту, справедливо
*    Недостатки: требует много памяти, медленнее (логирует каждый запрос)

### 4. Sliding Window Counter

*    Гибридный подход между token bucket и sliding window log
*    Использует два временных окна (текущее и предыдущее)
*    Интерполирует количество запросов между окнами
*    Преимущества: память-эффективнее, быстрее, приемлемая точность
*    Недостатки: немного менее точен на границах окон

### 5. Distributed Rate Limiter

*    Использует внешнее хранилище (Redis, Memcached) для отслеживания лимитов
*    Позволяет синхронизировать лимиты между несколькими серверами
*    Преимущества: работает в микросервисной архитектуре
*    Недостатки: добавляет сетевую задержку, зависимость от внешнего сервиса

### 6. Thread-based Rate Limiter

*    Использует java.util.concurrent инструменты (Semaphore, ReentrantLock)
*    Управляет одновременным доступом потоков
*    Преимущества: просто встраивается в Java
*    Недостатки: работает только внутри одного процесса

### Какой выбрать?

*    Token Bucket - универсальный выбор для большинства случаев
*    Sliding Window Counter - хороший баланс между простотой и точностью
*    Redis-based - если нужна распределенная система
*    Thread-based (Semaphore) - если только нужно ограничить одновременные запросы в одном приложении 

### Выбор зависит от:

*    Нужна ли вам поддержка микросервисов
*    Требуемая точность контроля
*    Объем памяти, доступный на сервере
*    Требуемая производительность
