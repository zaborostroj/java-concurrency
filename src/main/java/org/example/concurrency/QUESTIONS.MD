## Основы Java Concurrency

### Что такое поток (Thread) и чем он отличается от процесса?
* Процесс - отдельная программа с собственной памятью
* Поток - часть процесса, использует его память

### Как создать поток в Java?
1. **Наследование от Thread**
    ```
    class MyThread extends Thread {
      public void run() {
          System.out.println("Hello");
      }
    }
    
    new MyThread().start();
    ```
2. **Реализация Runnable**
    ```
    Runnable task = () -> System.out.println("Hello");
    new Thread(task).start();
    ```
3. **Через ExecutorService (submit)**
    ```
    ExecutorService pool = Executors.newFixedThreadPool(2);
    pool.submit(() -> System.out.println("Hello"));
    pool.shutdown();
    ```
4. **FutureTask + execute**
    ```
    ExecutorService pool = Executors.newFixedThreadPool(2);
    
    FutureTask<String> task = new FutureTask<>(() -> "Hello");
    pool.execute(task);
    
    String result = task.get();
    pool.shutdown();
    ```
5. **CompletableFuture.supplyAsync**
    ```
    CompletableFuture<String> future =
    CompletableFuture.supplyAsync(() -> "Hello");
    
    future.thenAccept(System.out::println);
    ```
6. **@Async (Spring)**
    ```
    @Async
    public void doWork() {
    System.out.println("Hello");
    }
    ```

### Чем Callable отличается от Runnable?
    ```
    Runnable - void run()
    Callable - T call() throws Exception
    ```

### Что такое synchronized и какие есть варианты его использования?
`Synchronized` - это механизм взаимного исключения (mutual exclusion) в Java.
Он гарантирует, что только один поток выполняет критическую секцию кода для одного объекта (монитора).

**Варианты использования**
1. **Синхронизированный метод (экземпляра)**

    Блокирует текущий объект (this)
    ```
    
    public synchronized void increment() {
        count++;
    }
    ```
2. **Синхронизированный статический метод**
    
    Блокирует класс (ClassName.class)
    ```
    public static synchronized void increment() {
        count++;
    }
    ```
3. **Синхронизированный блок**

    Можно явно указать объект для блокировки
    ```
    synchronized (lock) {
        count++;
    }
    ```

4. **Блок на this**
    ```
    synchronized (this) {
    count++;
    }
    ```

5. **Блок на классе**
    ```
    synchronized (MyClass.class) {
    count++;
    }
    ```

### Что такое monitor lock?
Monitor lock (монитор) - это встроенный механизм блокировки в Java, привязанный к каждому объекту.
Когда поток входит в synchronized, он захватывает монитор объекта, и другие потоки не могут войти в синхронизированный блок с тем же объектом.

Используется в synchronized методах и блоках, wait() / notify() / notifyAll().

### Что такое Reentrancy?
Это свойство блокировки, при котором тот же поток может повторно захватить уже удерживаемый им lock, не блокируясь.

    public synchronized void methodA() {
        methodB(); // тот же поток снова входит в synchronized
    }
    
    public synchronized void methodB() {
        // выполнится без блокировки
    }

Поток уже держит монитор this, поэтому повторный вход разрешён.

У каждого lock есть счётчик захватов (hold count)
Нужно столько же unlock (выходов), сколько было входов
В Java synchronized - реентерабельный

### Что такое Java Memory Model (JMM)?

* https://habr.com/en/articles/685518/
* https://jenkov.com/tutorials/java-concurrency/java-memory-model.html

В современном мире код часто выполняется не в том порядке, в котором он был написан в программе. Он часто переупорядочивается на уровне:
* Компилятора байткода (в частности, javac)
* Компилятора машинного кода (в частности, JIT компилятор HotSpot C1/C2). Например, среди компиляторов широко распространена такая оптимизация как Instruction scheduling
* Процессора. Например, в мире процессоров широко распространены такие практики как Out-of-order execution, Branch Prediction + Speculation, Prefetching, а также многие другие

Также в современных процессорах каждое ядро имеет собственный локальный кэш, который не видим другим ядрам. Более того, записи могут удерживаться в регистрах процессора, а не сбрасываться в память. Это ведет к тому, что поток может не видеть изменений, сделанных из других потоков.

Все эти оптимизации делаются с целью повысить производительность программ.

Есть хорошие новости, и плохие. Начнем с хорошей:

* Java дает гарантию as-if-serial выполнения кода - вне зависимости от используемой JDK итоговый результат выполнения будет не отличим от такого порядка, как если бы действия выполнялись действительно последовательно согласно порядку в коде
* Процессоры тоже делают только такие переупорядочивания, которые не изменят итогового результата выполнения инструкций
* Процессоры имеют Cache Coherence механизм, который гарантирует консистентность данных среди локальных кэшей: как только значение попадает в локальный кэш одного ядра, оно будет видно всем остальным ядрам

Теперь плохие новости:

* Java дает as-if-serial гарантию только для единственного треда в изоляции. Это означает, что в многопоточной программе при работе с shared данными мы можем не увидеть записи там, где полагаемся на порядок выполнения действий в коде другого треда. Другими словами, для первого треда в изоляции валидно переупорядочивать инструкции местами, если это не повлияет на его результат выполнения, но переупорядочивание может повлиять на другие треды
* Процессор также дает гарантию только для единственного ядра в изоляции
* Cache Coherence действительно гарантирует чтение актуальных значений, но пропагация записи происходит не мгновенно, а с некоторой задержкой

Мы поняли, что as-if-serial семантики недостаточно для многопоточных программ. Почему же не распространить as-if-serial гарантию на всю программу и ядра процессора? Ответ простой - это сильно ударило бы по производительности программ или процессора.

Модель памяти делает легальными многие оптимизации компилятора, JVM и процессора, но в то же время закрепляет условия, при которых программа будет вести себя корректно в многопоточной среде даже в присутствие оптимизаций. Таким образом, модель памяти:

* Разрешает выполнение различных оптимизаций компилятора, JVM или процессора
* Строго закрепляет условия, при которых программа считается правильно синхронизированной, и закрепляет поведение правильно синхронизированных программ
* Описывает отношение между высокоуровневым кодом и памятью
* Является trade-off между строгостью исполнения кода и возможными оптимизациями

Так вот, Java имеет свою модель памяти под названием Java Memory Model (JMM). По умолчанию JMM разрешает любые переупорядочивания и не гарантирует видимости изменений. Однако при выполнении определенных условий нам гарантируется порядок действий, консистентный с порядком в коде, а также видимость всех изменений. Таким образом, JMM позволяет нам писать программы, которые будут полностью корректно работать среди множества различных имплементаций JDK и микро-архитектур процессоров, в то же время сохраняя преимущества оптимизаций.

В многопоточной программе нам важно знать ответы на следующие вопросы:

1. Как сохраняется порядок программы при работе с памятью?
2. Валиден ли наблюдаемый memory order?

Дать ответ на каждый из вопросов - это и есть задача модели памяти. Java Memory Model разрешает все возможные переупорядочивания в отсутствие синхронизации, поэтому ответ на эти вопросы такой:

1. Если программа не синхронизирована, то разрешены все переупорядочивания. Если программа правильно синхронизирована, запрещены все переупорядочивания
2. Если программа не синхронизирована, то memory order, неконсистентный с program order, валиден с точки зрения JMM. Если программа правильно синхронизирована, то валиден только консистентный порядок

### Что делает ключевое слово volatile?

`volatile` гарантирует, что изменения переменной сразу видны всем потокам. volatile переменные всегда читатся/пишутся из общей памяти, без кэша, но без блокировок.

**Что обеспечивает:**
* Видимость (visibility)
* Запрещает опасные reordering вокруг переменной

**Чего не обеспечивает:**
* Не даёт атомарность
```
volatile int count = 0;
count++; // НЕ потокобезопасно
```
Когда использовать:
* Флаги (stop, ready)
* Простые состояния без сложной логики

### В чем разница между volatile и synchronized?

`volatile`
* Гарантирует видимость
* Нет взаимного исключения
* Нет атомарности
* Быстрее (без блокировок)

`synchronized`
* Видимость
* Взаимное исключение (mutex)
* Гарантирует атомарность внутри блока
* Медленнее (есть блокировка)

Когда что использовать
volatile - флаги, простые состояния
synchronized - когда есть критическая секция / изменение данных

### Что такое happens-before?

Happens-before - правило в Java Memory Model (JMM), которое определяет порядок действий между потоками, чтобы изменения одного потока были гарантированно видны другому.

Основные правила happens-before
1. **Программа - поток**
    
    Внутри одного потока инструкции выполняются в порядке кода (программный порядок)
2. **start()**
    ``` 
    Thread t = new Thread(r);
    t.start();   // Все изменения до start() видны новому потоку```
3. **join()**

    `t.join();    // Все изменения в потоке t видны после join()`
4. **Volatile**

    `volatile write - volatile read`
    Запись в volatile видна всем потокам, которые читают эту переменную
5. **Synchronized**
    ```
    unlock(monitor) - lock(monitor)
    ```
    Все изменения до выхода из synchronized блокa видны в блоке, который захватывает тот же монитор
6. **Thread termination**

    Поток завершён - любые последующие действия других потоков видят изменения этого потока

### Что такое false sharing?

**False sharing** - это проблема многопоточности, когда потоки конкурируют за разные переменные, которые находятся в одной кеш-линиии CPU.

Хотя переменные логически независимы, они физически делят кэш, и изменение одной переменной сбрасывает кэш другой, вызывая лишние задержки.

```
class Counter {
    volatile long a = 0;
    volatile long b = 0; // в одной кеш-линии с a
}
```
* Поток 1 обновляет a
* Поток 2 обновляет b
* Из-за кеш-линии CPU они сбрасывают друг другу кэш - падение производительности

**Как избежать:**
* Выравнивание переменных (padding)
* Использовать @Contended (Java 8+)
* Размещать часто изменяемые переменные в разных объектах


## Locks и конкуррентные примитивы

### Чем ReentrantLock отличается от synchronized?

| Особенность              | `synchronized`      | `ReentrantLock`                     |
|--------------------------|---------------------|-------------------------------------|
| Блокировка/разблокировка | Автоматическая      | Явная (`lock()` / `unlock()`)       |
| Таймаут захвата          | Нет                 | Да (`tryLock(timeout)`)             |
| Прерывание потоков       | Нет                 | Да (`lockInterruptibly()`)          |
| Проверка состояния       | Нет                 | Да (`isLocked()`, `getHoldCount()`) |
| Производительность       | Простая, стабильная | Быстрее при высокой конкуренции     |

### Что такое fairness (справедливые блокировки)?

**Fairness (справедливость)** - это политика выдачи блокировки потокам в порядке очереди (FIFO).

`Lock lock = new ReentrantLock(true); // fair`
* Fair lock (true) - поток получает lock в порядке ожидания
* Non-fair (false, по умолчанию) - lock может “перепрыгнуть очередь” (кто быстрее - тот и получил)
* Fair - меньше starvation (голодания потоков)
* Хуже производительность

### Что такое tryLock и зачем он нужен?
**tryLock()** - это попытка захватить блокировку без ожидания.

Если lock свободен - сразу захватывает и возвращает true
Если занят - сразу возвращает false (не блокируется поток)

Пример:
```
if (lock.tryLock()) {
    try {
        // работа с ресурсом
    } finally {
        lock.unlock();
    }
} else {
    // не удалось захватить lock
}
```
Есть вариант с таймаутом:
`lock.tryLock(2, TimeUnit.SECONDS);`

Зачем нужен:
* избегать блокировки потока (не зависает в ожидании)
* реализовать попытки с повтором
* снижать риск дедлоков

### Что такое ReadWriteLock?

**ReadWriteLock** - это механизм синхронизации с двумя типами блокировок:

**read lock** - для чтения (можно нескольким потокам одновременно)
**write lock** - для записи (только один поток, эксклюзивно)

### Что такое CAS (Compare-And-Swap)?

**CAS (Compare-And-Swap)** - это атомарная операция:
* сравнивает значение с ожидаемым
* если совпадает - заменяет на новое
* если нет - ничего не делает

**Зачем нужен:**
* основа lock-free алгоритмов
* используется в Atomic* классах
* быстрее, чем блокировки при низкой конкуренции

Возможна проблема ABA - это ситуация в CAS, когда значение изменилось, но CAS этого “не заметил”.

**Сценарий:**
* поток 1 читает значение A
* поток 2 меняет A → B → A
* поток 1 делает CAS (ожидал A) → успешно, хотя значение уже менялось

**Проблема:**
* CAS видит только текущее значение (A), но не знает, что оно уже было изменено

**Решение:**
* Использовать версионирование

### Что такое Atomic классы (AtomicInteger, AtomicReference)?

**Atomic классы** - это классы из java.util.concurrent.atomic, которые обеспечивают атомарные операции без блокировок.
* потокобезопасность без блокировок
* выше производительность при низкой и средней конкуренции
* основа lock-free структур данных

### Чем Atomic отличается от synchronized?

| Особенность        | `Atomic`                     | `synchronized`        |
|--------------------|------------------------------|-----------------------|
| Тип                | Без блокировок (lock-free)   | Блокировка (monitor)  |
| Механизм           | CAS                          | Монитор (lock)        |
| Блокирует поток    | Нет                          | Да                    |
| Производительность | Лучше при низкой конкуренции | Стабильна, но тяжелее |
| Гибкость           | Только для простых операций  | Любая логика          |

* **Atomic** лучше для простых операций типа счетчиков
* **synchronized** лучше для сложной логики и нескольких действий в одной критической секции


## Concurrent Collections

### Чем HashMap отличается от ConcurrentHashMap?
| Критерий            | HashMap                                    | ConcurrentHashMap                    |
|---------------------|--------------------------------------------|--------------------------------------|
| Потокобезопасность  | - нет                                      | + да                                 |
| Блокировки          | - отсутствуют                              | + на уровне сегментов/бакетов        |
| Производительность  | + быстрее в одном потоке                   | + лучше при многопоточности          |
| Null ключи/значения | + допускает null                           | - null запрещён                      |
| Итерация            | - возможен ConcurrentModificationException | + слабосогласованные, без исключений |

### Как работает ConcurrentHashMap (до Java 8 и после)?

| Версия         | Как работает                                                             |
|----------------|--------------------------------------------------------------------------|
| До Java 8      | - деление на сегменты (Segment[])                                        |
|                | - каждый сегмент = отдельный lock                                        |
|                | - операции блокируют только один сегмент                                 |
|                | - фиксированное число сегментов                                          |
|                | - внутри сегмента почти как HashMap                                      |
| После Java 8   | - сегментов нет, единый массив Node[]                                    |
|                | - используется CAS (compare-and-swap) + synchronized при необходимости   |
|                | - блокировки на уровне отдельных бакетов                                 |
|                | - при коллизиях: список -> красно-чёрное дерево                          |
|                | - более высокая масштабируемость и гибкость                              |

### Почему нельзя использовать обычный ArrayList в многопоточке?

* ArrayList не потокобезопасен — нет синхронизации
* При одновременных изменениях возникают гонки данных (race conditions)
* Возможна потеря элементов или перезапись данных
* Во время resize (увеличения массива) структура может сломаться
* Итерация может привести к ConcurrentModificationException

Если нужен аналог - Collections.synchronizedList(...) или CopyOnWriteArrayList

### Чем CopyOnWriteArrayList отличается от Collections.synchronizedList()?

**Подход к синхронизации**
* CopyOnWriteArrayList: при каждом изменении создаётся новая копия массива
* Collections.synchronizedList: все операции обёрнуты в synchronized (одна блокировка)

**Чтение**
* CopyOnWriteArrayList: очень быстрое, без блокировок
* synchronizedList: с блокировкой

**Запись (add/remove)**
* CopyOnWriteArrayList: дорогая операция (копирование массива)
* synchronizedList: дешевле, но блокирует всех

**Итерация**
* CopyOnWriteArrayList: безопасна, без ConcurrentModificationException, работает со snapshot
* synchronizedList: нужно вручную синхронизировать при итерации

**Когда использовать**
* CopyOnWriteArrayList: когда много чтений и мало записей
* synchronizedList: когда запись происходит часто

### Что такое BlockingQueue и какие есть реализации?

**BlockingQueue** — это потокобезопасная очередь, которая умеет блокировать потоки при операциях вставки и извлечения
**Как работает**
* при put() — поток ждёт, если очередь заполнена
* при take() — поток ждёт, если очередь пуста
* не нужно вручную писать wait/notify

**Основные реализации**
1. **ArrayBlockingQueue**
   * ограниченный размер (bounded)
   * основана на массиве
2. LinkedBlockingQueue
   * может **быть ограниченной** или почти неограниченной
   * основана на связном списке
3. **PriorityBlockingQueue**
   * элементы упорядочены по приоритету
   * не ограничена
4. **DelayQueue**
   * элементы доступны только после задержки
5. **SynchronousQueue**
   * не хранит элементы вообще
   * каждый put ждёт take
6. **LinkedTransferQueue**
   * расширенная неблокирующая очередь
   * поддерживает передачу элемента напрямую потребителю

### Разница между ArrayBlockingQueue и LinkedBlockingQueue?

| Критерий            | ArrayBlockingQueue              | LinkedBlockingQueue         |
|---------------------|---------------------------------|-----------------------------|
| Структура           | массив                          | связный список              |
| Ограничение размера | всегда bounded                  | bounded или почти unbounded |
| Блокировки          | один lock                       | два lock (put и take)       |
| Производительность  | + лучше при высокой конкуренции | + лучше масштабируется      |
| Память              | + меньше overhead               | - больше overhead (узлы)    |
| Предсказуемость     | + стабильнее latency            | - менее предсказуемая       |
                 

## Executors и Thread Pools

### Что такое ExecutorService?

* **ExecutorService** — это интерфейс в Java для управления потоками, который **отделяет создание и управление потоками от их использования**.

* **Зачем нужен**

    * не нужно вручную создавать Thread
    * можно повторно использовать пул потоков
    * упрощает управление задачами, очередями и завершением работы

* **Основные возможности**

    * `submit()` / `execute()` — запуск задач
    * `shutdown()` / `shutdownNow()` — корректное завершение работы
    * `invokeAll()` / `invokeAny()` — выполнение коллекции задач с возвратом результатов
    * поддержка Future для получения результата асинхронной задачи

* **Типичные реализации**

    * ThreadPoolExecutor
    * ScheduledThreadPoolExecutor
    * через фабрики Executors.newFixedThreadPool, newCachedThreadPool, newSingleThreadExecutor и др.

### Чем execute() отличается от submit()?

* **execute()**

    * метод из интерфейса `Executor`
    * запускает задачу `Runnable`
    * **не возвращает результат**
    * при ошибке выбрасывает unchecked exception сразу

* **submit()**

    * метод из `ExecutorService`
    * может принимать `Runnable` или `Callable`
    * **возвращает Future** для получения результата или проверки завершения
    * исключения внутри задачи можно получить через `Future.get()`, не ломая поток сразу

### Какие типы ThreadPool существуют (Fixed, Cached, Scheduled)?

* **FixedThreadPool**

    * фиксированное число потоков
    * новые задачи ждут в очереди, если все потоки заняты
    * хорошо для стабильной нагрузки

* **CachedThreadPool**

    * динамическое число потоков
    * создаёт новые потоки при необходимости, переиспользует свободные
    * хорошо для краткоживущих и большого количества коротких задач

* **ScheduledThreadPool**

    * пул с планировщиком задач
    * поддерживает задержку (`schedule`) и периодическое выполнение (`scheduleAtFixedRate`, `scheduleWithFixedDelay`)

* **SingleThreadExecutor**

    * один поток
    * задачи выполняются строго последовательно
    * удобно для гарантированной последовательности выполнения

* **WorkStealingPool (Java 8+)**

    * пул с work-stealing алгоритмом
    * потоки берут задачи из очередей друг друга при простое
    * хорошо для параллельных задач, разбиваемых на мелкие части

### Как работает ThreadPoolExecutor? (corePoolSize, maxPoolSize, queue)

**ThreadPoolExecutor** — это основной класс реализации пула потоков. Его работа зависит от трёх ключевых параметров:

* **corePoolSize**

    * минимальное число потоков, которые **держатся всегда**, даже если они простаивают
    * новые задачи сначала пытаются использовать эти потоки

* **maximumPoolSize**

    * максимальное число потоков, которые могут быть созданы при высокой нагрузке
    * если все core-потоки заняты и очередь заполнена, создаются новые потоки до maxPoolSize

* **workQueue (очередь задач)**

    * задачи, которые не могут сразу выполняться, помещаются в очередь
    * тип очереди влияет на поведение пула:

        * **LinkedBlockingQueue** — почти неограниченная очередь, новые потоки почти не создаются
        * **ArrayBlockingQueue** — фиксированная очередь, новые потоки создаются до maxPoolSize
        * **SynchronousQueue** — не хранит элементы, каждый put ждёт take, новые потоки создаются сразу

**Алгоритм работы:**

1. Если количество потоков < corePoolSize → создаётся новый поток для задачи.
2. Иначе → пытается добавить задачу в очередь.
3. Если очередь полна и количество потоков < maxPoolSize → создаётся новый поток.
4. Если очередь полна и достигнут maxPoolSize → задача отклоняется (RejectedExecutionHandler).

Правильное завершение ThreadPool должно быть **корректным и безопасным**, чтобы не потерять задачи и не оставить висящие потоки. В Java есть два основных метода:

1. **shutdown()**

    * пул **не принимает новые задачи**, но выполняет все уже добавленные в очередь
    * после выполнения всех задач поток автоматически завершится
    * можно проверить завершение через `awaitTermination(timeout, unit)`

2. **shutdownNow()**

    * пул пытается **немедленно остановить все активные задачи**
    * возвращает список задач, которые ещё не начали выполняться
    * активные потоки прерываются через `Thread.interrupt()`
    * следует быть осторожным: задачи должны корректно обрабатывать прерывания

**Рекомендации:**

* Обычно сначала вызывают `shutdown()`, ждут завершения через `awaitTermination()`.
* Если после таймаута пул всё ещё не завершился → `shutdownNow()`.
* Всегда обрабатывать InterruptedException в задачах, если используете shutdownNow().


## Coordination & Synchronizers

### Что такое CountDownLatch и где используется?

* **CountDownLatch** — это синхронизатор потоков в Java, который позволяет одному или нескольким потокам **ждать, пока другие потоки завершат свою работу**.

* **Как работает**

    * создаётся с числом `count`
    * вызывая `countDown()`, поток уменьшает счётчик на 1
    * поток, вызвавший `await()`, блокируется до тех пор, пока счётчик не станет 0

* **Использование**

    * ожидание завершения группы потоков перед продолжением работы
    * запуск нескольких потоков одновременно после подготовки (с latch с count = 1)
    * синхронизация этапов в тестах или сложных алгоритмах

* **Пример сценариев**

    * загрузка ресурсов перед стартом приложения
    * выполнение параллельных задач и ожидание их результата перед объединением
    * интеграционные и нагрузочные тесты

### Чем CyclicBarrier отличается от CountDownLatch?

* **CountDownLatch**

    * одноразовый: после того как счётчик дошёл до нуля, его **нельзя перезапустить**
    * один или несколько потоков ждут, пока другие потоки завершат работу
    * используется для **ожидания завершения группы задач**

* **CyclicBarrier**

    * многоразовый: после срабатывания **можно использовать снова**
    * группа потоков ждёт друг друга, и как только все достигли барьера, все продолжают работу
    * используется для **синхронизации этапов между потоками**, например шагов в вычислениях

### Что такое Semaphore?

* **Semaphore** — это счётчик разрешений для потоков, который управляет доступом к ограниченному ресурсу.

* **Как работает**

    * создаётся с числом `permits` — количества потоков, которые могут одновременно получить доступ
    * `acquire()` — поток запрашивает разрешение; если нет свободных, блокируется
    * `release()` — поток освобождает разрешение, позволяя другому потоку продолжить

* **Использование**

    * ограничение числа одновременно работающих потоков с ресурсом
    * контроль доступа к пулам соединений, базам данных, файлам
    * реализация пропускной способности или throttling

* **Особенности**

    * можно использовать fair-семафор (`new Semaphore(permits, true)`), чтобы разрешения выдавались в порядке ожидания
    * может использоваться для сигнализации между потоками, как простой “счётчик событий”

### Что такое Phaser?

* **Phaser** — это гибкий синхронизатор потоков в Java, который объединяет идеи **CountDownLatch** и **CyclicBarrier**, но более универсален.

* **Как работает**

    * потоки регистрируются в фазах с помощью `register()`
    * каждый поток вызывает `arrive()` или `arriveAndAwaitAdvance()` после выполнения своей части работы
    * как только все зарегистрированные потоки дошли до барьера, **фаза завершается и счётчик обновляется**, переходя к следующей фазе
    * поддерживает **многоразовое использование** (циклический барьер)

* **Преимущества**

    * динамическое добавление или снятие потоков (`register()`, `bulkRegister()`, `arriveAndDeregister()`)
    * управление многими фазами работы одновременно
    * можно использовать для сложных параллельных вычислений с этапами

* **Использование**

    * пошаговые параллельные вычисления (например, симуляции, игры, расчёты на сетке)
    * когда количество участников меняется на разных фазах
    * замена сочетания CountDownLatch + CyclicBarrier в динамических сценариях

### Что такое Exchanger?

* **Exchanger** — это синхронизатор потоков в Java, который позволяет **двум потокам обмениваться данными напрямую**.

* **Как работает**

    * создаётся объект `Exchanger<T>`
    * один поток вызывает `exchange(data1)`
    * второй поток вызывает `exchange(data2)`
    * когда оба потока достигли точки обмена, **данные обмениваются между ними** и оба продолжают работу

* **Особенности**

    * обмен происходит **парами**
    * блокирует поток до тех пор, пока второй поток не придёт
    * удобен для реализации "поставщик-потребитель" без очередей

* **Использование**

    * обмен данными между потоками в пайплайне
    * буферизация, когда два потока синхронизируются на каждом шаге
    * совместная обработка данных двумя потоками

## Deadlocks, Race Conditions и проблемы многопоточности

### Что такое race condition?

Race condition – это ситуация в многопоточной программе, когда результат выполнения зависит от порядка выполнения потоков и может быть непредсказуемым.

### Что такое deadlock? Как его избежать?

Deadlock – это ситуация, когда два или более потоков навсегда блокируют друг друга, ожидая ресурсы, занятые друг другом.

Как избежать:

* Использовать единый порядок захвата ресурсов.
* Ограничивать время ожидания (timeout).
* Использовать высокоуровневые синхронизации (например, `Lock.tryLock()` вместо `synchronized`).
* Минимизировать количество захватываемых ресурсов одновременно.

### Что такое livelock и starvation?

* **Livelock** – ситуация, когда потоки активно выполняются, но не могут продвинуться вперед, постоянно реагируя друг на друга.
* **Starvation** – ситуация, когда поток долго не получает доступ к ресурсу из-за того, что другие потоки постоянно его блокируют.

### Как обнаружить deadlock в JVM?

* Использовать `jstack <pid>` для получения дампа потоков и искать состояние `BLOCKED` с взаимной блокировкой.
* В коде применять `ThreadMXBean` и метод `findDeadlockedThreads()`.
* Использовать профилировщики/мониторы потоков (VisualVM, JConsole) для визуализации блокировок.


## Design Patterns в многопоточности

### Как сделать Singleton thread-safe?

1. **Init on-demand holder**
    ```
    public class InitOnDemandHolderSingleton {
        private static class Holder {
            private static final InitOnDemandHolderSingleton INSTANCE = new InitOnDemandHolderSingleton();
        }
    
        private InitOnDemandHolderSingleton() {}
    
        public static InitOnDemandHolderSingleton getInstance() {
            return Holder.INSTANCE;
        }
    
        public String sayHello() {
            return "Hello from " + getClass().getSimpleName() + "@" + System.identityHashCode(this);
        }
    }
    ```
    Class loader создаст класс-обёртку `InitOnDemandHolderSingleton`, но его внутренний статический класс `Holder` будет создан только при первом обращении к методу `getInstance()`. 

2. **Double-checked locking**
    ```
    public class DoubleCheckLockingSingleton {
    private static volatile DoubleCheckLockingSingleton INSTANCE;

    private DoubleCheckLockingSingleton() {}

    public static DoubleCheckLockingSingleton getInstance() {
        DoubleCheckLockingSingleton instance = INSTANCE;
        if (instance == null) {
            synchronized (DoubleCheckLockingSingleton.class) {
                instance = INSTANCE;
                if (instance == null) {
                    INSTANCE = new DoubleCheckLockingSingleton();
                    instance = INSTANCE;
                }
            }
        }
        return instance;
    }
    ```
   Создание `DoubleCheckLockingSingleton`состоит из трёх этапов:
   1. Выделение памяти
   2. вызов конструктора
   3. запись ссылки на объект в переменную INSTANCE
    
   JVM может сделать reordering для операций 2 и 3:
   * поток А входит в synchronized блок и присваивает адрес памяти переменной INSTANCE
   * поток B вызывает `getInstance()`, видит `INSTANCE != null` и сразу возвращает ссылку на объект
   * поток B начинает работу с неполностью инициализированным объектом
    
   Именно поэтому `INSTANCE` должна быть `volatile` - это защитит от переупорядочивания операций.

3. **Enum**
    ```
    public enum EnumSingleton {
        INSTANCE("EnumSingleton info");
    
        private String info;
    
        private EnumSingleton(String info) {
            this.info = info;
        }
    
        public String getInfo() {
            return info;
        }
    
        public static EnumSingleton getInstance() {
            return INSTANCE;
        }
    
        public String sayHello() {
            return "Hello from " + getClass().getSimpleName() + "@" + System.identityHashCode(this);
        }
    }
    ```
   
### Что такое Double-Checked Locking и почему он был сломан до Java 5?

**Double-Checked Locking (DCL)** – это паттерн ленивой инициализации Singleton, где проверка `instance == null` делается дважды:

* * первый раз без блокировки (для скорости)
* * второй раз внутри `synchronized` (для безопасности)

Проблема до Java 5:

* * Из-за особенностей **модели памяти (JMM)** могла происходить **переупорядочивание инструкций**
* * Объект мог быть присвоен в `instance` до полной инициализации
* * Другой поток видел **не до конца созданный объект**

Почему исправили:

* * Начиная с Java 5, ключевое слово `volatile` гарантирует:

    - - отсутствие переупорядочивания
    - * видимость изменений между потоками

### Что такое Producer-Consumer pattern?

**Producer-Consumer pattern** – это шаблон многопоточности, где:

* **Producer (производитель)** создаёт данные
* **Consumer (потребитель)** обрабатывает данные
* Между ними используется общая очередь (buffer)

Как работает:

* Producer кладёт задачи в очередь
* Consumer берёт задачи из очереди
* Если очередь пуста – consumer ждёт
* Если очередь заполнена – producer ждёт

В Java обычно реализуется через:

* `BlockingQueue` (например, `ArrayBlockingQueue`, `LinkedBlockingQueue`)
* или через `wait()/notify()`

### Что такое Immutable objects и зачем они важны для concurrency?

**Immutable object** – это объект, состояние которого нельзя изменить после создания.

Признаки:

* Все поля `final`
* Нет сеттеров
* Поля не изменяются после конструктора
* Класс часто делают `final`

Почему важны для concurrency:

* Потокобезопасны по умолчанию (нет состояния, которое можно изменить)
* Не требуют синхронизации
* Нет race condition
* Безопасно шарить между потоками

### Как реализовать thread-safe cache?

Основные способы сделать thread-safe cache в Java:

1. **ConcurrentHashMap (самый простой вариант)**

    ```java
    Map<Key, Value> cache = new ConcurrentHashMap<>();
    
    public Value get(Key key) {
        return cache.computeIfAbsent(key, k -> load(k));
    }
    ```
    
    * Потокобезопасно из коробки
    * Хорошая производительность

2. **Синхронизация (менее эффективно)**

    ```java
    private final Map<Key, Value> cache = new HashMap<>();
    
    public synchronized Value get(Key key) {
        return cache.computeIfAbsent(key, k -> load(k));
    }
    ```
    
    * Просто
    * Узкое место из-за блокировки

3. **ReadWriteLock**

    ```java
    private final Map<Key, Value> cache = new HashMap<>();
    private final ReadWriteLock lock = new ReentrantReadWriteLock();
    
    public Value get(Key key) {
        lock.readLock().lock();
        try {
            if (cache.containsKey(key)) {
                return cache.get(key);
            }
        } finally {
            lock.readLock().unlock();
        }
    
        lock.writeLock().lock();
        try {
            return cache.computeIfAbsent(key, k -> load(k));
        } finally {
            lock.writeLock().unlock();
        }
    }
    ```
    
    * Лучше при большом количестве чтений
    * Сложнее код

4. **Готовые решения**

    * Caffeine
    * Guava Cache


## Performance & Advanced Topics

### Почему synchronized может быть медленным? Какие оптимизации JVM существуют?

`synchronized` может быть медленным из-за:

* **Блокировок (monitor enter/exit)** – потоки могут ждать друг друга
* **Контекстных переключений** – при блокировке поток уходит в sleep и обратно
* **Контенции** – много потоков пытаются захватить один монитор
* **Перехода в kernel mode** при тяжёлых блокировках

Оптимизации JVM:

* **Biased locking (смещённая блокировка)**

    - оптимизация для случая, когда монитор использует один поток
    - практически без синхронизации

* **Lightweight locking (тонкие блокировки)**

    - через CAS без перехода в OS
    - быстрее, чем тяжёлые блокировки

* **Lock elision (устранение блокировок)**

    - JVM убирает `synchronized`, если он не нужен (escape analysis)

* **Lock coarsening**

    - объединение нескольких блокировок в одну

* **Spin-lock (активное ожидание)**

    - поток немного ждёт, не уходя в sleep

### Что такое biased locking, lightweight lock, heavyweight lock?

Это уровни оптимизации блокировок в JVM для `synchronized`:

**1. Biased locking (смещённая блокировка)**

* Оптимизация для случая, когда монитор захватывает один поток
* Блокировка “привязывается” к потоку и не требует синхронизации
* Нет CAS и почти нет накладных расходов
* При появлении другого потока происходит “отзыв” (revocation)

**2. Lightweight lock (лёгкая блокировка)**

* Использует CAS вместо тяжёлых mutex
* Потоки не блокируются сразу, сначала пытаются захватить lock
* Может использовать spin (активное ожидание)
* При высокой конкуренции может “раздуться”

**3. Heavyweight lock (тяжёлая блокировка)**

* Использует OS mutex (monitor)
* Потоки блокируются и переводятся в состояние ожидания
* Дорогие context switch и переходы в kernel mode

### Что такое ForkJoinPool и Work-Stealing?

**ForkJoinPool** – это пул потоков для выполнения задач, которые можно разбивать на подзадачи (подход *divide and conquer*).

* Задача делится (`fork`) на мелкие
* Результаты объединяются (`join`)
* Используется в `parallelStream`, `CompletableFuture`

**Work-Stealing** – механизм балансировки нагрузки:

* У каждого потока есть своя очередь задач
* Поток берёт задачи из своей очереди
* Если задачи закончились – “ворует” (steal) задачи у других потоков

Плюсы:
* Хорошая утилизация CPU
* Меньше простаивающих потоков
* Автоматический баланс нагрузки

### Как работает CompletableFuture?

**CompletableFuture** – это класс для асинхронных вычислений и композиции задач.

Как работает:

* Задача выполняется в другом потоке (обычно `ForkJoinPool`)
* Результат появится в будущем (future)
* Можно навешивать цепочку действий

У `CompletableFuture` много методов, их можно разделить на группы:

**1. Создание**

* `supplyAsync(Supplier)` – асинхронно возвращает результат
* `runAsync(Runnable)` – асинхронно выполняет задачу без результата
* `completedFuture(T value)` – уже завершённый future

**2. Преобразование / цепочки**

* `thenApply(Function)` – преобразует результат
* `thenAccept(Consumer)` – использует результат, ничего не возвращает
* `thenRun(Runnable)` – запускает действие после завершения, без результата
* `thenCompose(Function)` – flatMap, создаёт новый future из результата

**3. Комбинирование**

* `thenCombine(CompletableFuture, BiFunction)` – объединяет два future
* `thenAcceptBoth(CompletableFuture, BiConsumer)` – использует два результата
* `runAfterBoth(CompletableFuture, Runnable)` – действие после обоих

**4. Обработка ошибок**

* `exceptionally(Function)` – fallback при исключении
* `handle(BiFunction)` – результат + исключение
* `whenComplete(BiConsumer)` – наблюдение за завершением

**5. Завершение**

* `complete(T value)` – форсированно завершить
* `completeExceptionally(Throwable)` – завершить с исключением
* `get()`, `join()` – блокирующее получение результата

**6. Другие полезные**

* `allOf(CompletableFuture...)` – ждёт завершения всех
* `anyOf(CompletableFuture...)` – ждёт завершения хотя бы одного

`CompletableFuture` удобен для асинхронных цепочек, комбинирования и обработки ошибок без ручного управления потоками

### Разница между parallel streams и thread pools?

Основные различия:

**1. Parallel Streams**

* Высокоуровневый API для параллельной обработки коллекций (`stream().parallel()`)
* Использует **ForkJoinPool.commonPool()** по умолчанию
* Легко писать, мало кода
* Меньший контроль над потоками, размером пула и приоритетами
* Подходит для **CPU-bound** задач

**2. Thread Pools**

* Низкоуровневый инструмент (`ExecutorService`)
* Полный контроль над количеством потоков, политикой очередей, приоритетами
* Можно использовать для **CPU-bound и IO-bound** задач
* Нужно больше кода и управления
* Ручное завершение пула (`shutdown`)

### Что такое Reactive programming и чем отличается от классического multithreading?

**Reactive Programming** – это парадигма, где программы реагируют на поток данных и события, вместо того чтобы управлять потоками напрямую.

Особенности:

* * **Асинхронность**: задачи не блокируют потоки, обрабатываются по мере появления данных
* * **Обработчик событий**: операции выполняются в ответ на события (callbacks, signals)
* * **Потоки данных (streams)**: данные проходят через цепочку операторов (`map`, `filter`, `flatMap`)

**Отличия от классического multithreading**:

| Характеристика         | Multithreading                                | Reactive                                                  |
|------------------------|-----------------------------------------------|-----------------------------------------------------------|
| Управление потоками    | Явное, блокирующее (`Thread`, `synchronized`) | Неявное, неблокирующее (`Event loop`, `Schedulers`)       |
| Использование ресурсов | Поток = ресурс, блокировка дорого             | Потоки эффективно переиспользуются, блокировки минимальны |
| Подход к обработке     | Императивный: «делай шаг за шагом»            | Декларативный: «что делать с данными, когда они придут»   |
| Масштабируемость       | Ограничена количеством потоков                | Высокая, особенно для IO-bound задач                      |

Итог:

* * Reactive подходит для систем с большим количеством событий и IO
* * Multithreading лучше для CPU-bound задач с небольшим числом потоков

## Trick Questions

### Почему volatile не делает операции атомарными? Можно ли сделать ++i thread-safe с volatile?

`volatile` в Java обеспечивает только:

* * **Видимость** – если один поток записал значение, другие потоки увидят его сразу
* * **Запрет переупорядочивания** – операции с volatile не будут пересортированы компилятором или CPU

**Почему не атомарно:**

* Атомность означает, что операция выполняется **целиком**, без прерывания
* `volatile` не объединяет несколько шагов в одну атомарную операцию
    Например: `count++` → это три шага:

    1. прочитать значение
    2. увеличить
    3. записать обратно
* Между этими шагами другой поток может вмешаться → race condition

### Почему ConcurrentHashMap не допускает null?

`ConcurrentHashMap` не допускает `null` для ключей и значений по нескольким причинам:

1. **Неоднозначность при get/put**

   * `map.get(key)` возвращает `null`, если ключ отсутствует
   * Если разрешить `null` как значение, нельзя отличить **“ключа нет”** от **“значение null”**

2. **Проблемы с атомарными операциями**

   * Методы `computeIfAbsent`, `putIfAbsent`, `merge` и CAS-операции предполагают, что значение реально существует
   * `null` ломает логику атомарных обновлений

3. **Потокобезопасность**

   * При многопоточном доступе null может вызвать race condition или NullPointerException при вычислении/обновлении

### Что будет, если вызвать notify() без synchronized?

Если вызвать `notify()` без `synchronized`

* * JVM выбросит `IllegalMonitorStateException`
* * Потому что `notify()` должен вызываться только на мониторе объекта, который поток **захватил** через `synchronized`
* * Без захвата монитора нельзя уведомлять другие потоки, они не будут ждать правильно

### Чем wait() отличается от sleep()?

* **wait()**

    - Освобождает монитор (lock) объекта
    - Используется внутри `synchronized` блока
    - Ждёт уведомления (`notify()` / `notifyAll()`) или таймаута
    - Не блокирует весь поток, а только текущий монитор

* **sleep()**

    - Не освобождает монитор
    - Можно вызывать в любом месте
    - Просто приостанавливает поток на указанное время
    - Поток остаётся заблокированным на всех захваченных мониторах

### Почему Thread.stop() deprecated?

`Thread.stop()` устарел, потому что он **небезопасен**

* Прерывает поток **в любом месте**, сразу выбрасывая `ThreadDeath`
* Может оставить объекты в **неконсистентном состоянии**
* Мониторы (locks) не освобождаются корректно → возможны **deadlock**
* Прерывает операции, которые должны быть атомарными → **повреждение данных**

Безопасная альтернатива

* Использовать **флаг прерывания** (`volatile boolean` или `Thread.interrupt()`)
* Поток сам проверяет флаг и корректно завершает работу

### Как JVM гарантирует видимость final-полей?

JVM гарантирует видимость **final-полей** с помощью **специальных правил в Java Memory Model**

* * При создании объекта и завершении конструктора **любые final-поля “записываются” в память до публикации ссылки**
* * Другие потоки, получившие ссылку на объект, **видят корректные значения final-полей**, даже без синхронизации
* * Это работает даже если объект опубликован небезопасно (например, через обычную переменную без `volatile`)

Особенности

* * Гарантия распространяется **только на final-поля**
* * Если поле не final → для корректной видимости нужна синхронизация или `volatile`
